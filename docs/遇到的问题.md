# AI Secretary Demo 遇到的问题

## 1. 整段文本直接让 LLM 输出完整任务 JSON 不够稳定

### 问题现象

最初方案是把用户输入的整段自然语言一次性交给 LLM，让 LLM 直接返回完整任务 JSON。

例如用户输入：

```text
明天下午三点提醒我给王总发邮件，内容是项目方案已经更新；另外每周五上午十点通知团队群提交周报。收到李雷的消息后回复他：我下午会确认合同。
```

期望 LLM 一次返回多个结构化任务，包括动作类型、目标对象、内容、时间、置信度、是否需要确认等字段。

但实际测试中，这种方式不够稳定，主要表现为：

- 多个任务容易漏拆，整段话只生成一个任务。
- 任务之间的字段容易串扰，例如第一个任务的时间被带到第二个任务。
- `target.name` 容易把动作和内容一起吞进去，例如把“确认合同盖章”误识别成目标对象。
- 条件句、代词和上下文引用不稳定，例如“提醒他一次”容易把“他”直接作为目标。
- LLM 输出 JSON 虽然格式正确，但语义字段不一定可靠。
- 一旦输出结构偏差，Java 侧很难判断是解析错误还是业务数据异常。

### 原因分析

整段文本直接输出完整 JSON，本质上让 LLM 在一次生成里同时完成了太多事情：

- 判断句子边界。
- 判断任务边界。
- 判断条件块。
- 识别动作类型。
- 抽取目标、内容、时间等槽位。
- 生成调度结构。
- 处理代词和上下文。
- 输出符合 Java/Python 契约的 JSON。

这些子任务之间互相依赖，任何一个环节出错都会污染最终 JSON。尤其是中文办公口语里经常出现省略、代词、连续动作和条件触发，直接端到端生成会放大不确定性。

### 调整方案

将解析链路从：

```text
整段文本 -> LLM 直接吐完整任务 JSON
```

调整为：

```text
整段文本
  -> 分句/分条件块
  -> 识别动作类型
  -> 抽取槽位
  -> 校验/修复
  -> 输出任务
```

### 新链路说明

#### 1. 分句/分条件块

先把用户输入拆成更小的语义单元。

拆分依据包括：

- 标点：句号、分号、换行。
- 连接词：另外、然后、并且、同时。
- 条件触发结构：如果、当、收到、到点后、超过多久等。

目标是先确定“哪些片段各自对应一个可执行任务”，避免一段话里多个动作互相干扰。

#### 2. 识别动作类型

对每个语义片段单独判断动作类型。

当前动作类型包括：

- `send_email`
- `send_message`
- `reply_message`
- `reminder`
- `create_todo`
- `schedule_task`

这样可以先确定任务大类，再决定后续槽位应该怎么抽。

例如：

- “发邮件”优先抽取收件人、邮件内容、发送时间。
- “提醒我”优先抽取提醒对象、提醒内容、提醒时间。
- “收到消息后回复”优先抽取消息来源、回复内容和触发条件。

#### 3. 抽取槽位

在动作类型确定后，再抽取对应槽位。

核心槽位包括：

- `target`：接收对象或执行对象。
- `content`：任务正文、邮件正文、消息正文或提醒内容。
- `schedule`：时间计划。
- `priority`：优先级。
- `requiresConfirmation`：是否需要用户确认。
- `sourceSentence`：原始来源片段。

这样可以减少无关字段互相污染。例如 `send_email` 关注收件人和邮件正文，`reminder` 关注提醒内容和时间，不强行套同一套抽取逻辑。

#### 4. 校验/修复

LLM 或规则解析输出后，还要做确定性校验和修复。

重点校验包括：

- `actionType` 是否在枚举范围内。
- `scheduleType` 是否为 `none` / `once` / `recurring`。
- `runAt` 是否符合 ISO-8601。
- `cron` 是否只用于周期任务。
- `target.name` 是否错误包含动作词或任务内容。
- 高风险动作是否设置 `requiresConfirmation=true`。
- 代词“他/她/他们/对方”是否被错误当成具体目标。

如果发现边界错误，则通过后处理修复：

- 把误进 `target.name` 的动作短语移回 `content`。
- 无明确目标时改为 `unknown` 或 `self`。
- 条件产生的人群改为 `dynamic`。
- 无法安全判断时保留警告，交给用户确认。

#### 5. 输出任务

最终输出稳定的任务结构，供 Java 服务保存、预览、确认、调度和模拟执行。

输出对象保持为：

- `TaskPlan`
- `TaskAction`
- `TaskTarget`
- `TaskSchedule`

Java 侧不直接关心解析过程，只消费稳定契约。

### 调整后的收益

- 多任务拆分更清晰，减少漏拆和合并。
- 字段边界更稳定，尤其是目标对象和任务内容。
- LLM 输出不再被完全信任，可以通过规则校验兜底。
- Java/Python 职责边界更清楚：Python 做语义解析和修复，Java 做业务编排。
- 后续排查问题更容易定位，是分句错、动作类型错、槽位错，还是后处理修复不足。
- 更容易积累回归测试样例，对单个环节做定向优化。

### 示例对比

原始输入：

```text
提醒我给李雷确认合同盖章。
```

不稳定结果可能是：

```json
{
  "action_type": "reminder",
  "target": {
    "target_type": "user",
    "name": "李雷确认合同盖章"
  },
  "content": ""
}
```

修复后的期望结果：

```json
{
  "action_type": "reminder",
  "target": {
    "target_type": "user",
    "name": "李雷"
  },
  "content": "确认合同盖章"
}
```

### 后续注意点

- 不要只靠 prompt 修复所有问题，稳定可复现的问题优先放进后处理规则。
- 每新增一种动作类型，都要明确对应槽位和校验规则。
- 条件块和动态目标要保留原始上下文，不要过早丢失来源句子。
- 高风险动作必须保留确认机制，不能因为解析置信度高就自动执行。
- 每次修复解析问题后，应补充至少一个回归样例。

## 2. fallback 规则解析器不应承担复杂长文本语义解析

### 问题现象

在 OpenRouter Key 不可用或 LLM 调用失败时，Python 服务会回退到本地规则解析器。早期为了让 demo “始终有结果”，fallback 会继续尝试拆解长文本。

真实测试中发现，这会带来更大的误导：

- 长文本会被拆出大量看似结构化、实际不可靠的任务。
- `target.name` 容易出现“我”“一次”“大家”等泛指词，或者把动作短语误当对象。
- 条件句如“如果...就...”没有明确时间，却被页面展示为“立即/待确认”，容易让用户以为可以马上执行。
- 邮件主题、正文、执行策略混在一起时，规则很容易截断错误。
- 由于结果看起来是正常任务卡片，用户不容易意识到当前不是 LLM 语义解析结果。

### 原因分析

fallback 的定位应该是保底，而不是替代 LLM。

规则解析器适合处理边界清晰的简单句：

```text
明天上午九点提醒我整理项目进度。
每周五上午十点通知研发群提交周报。
```

但不适合处理复杂办公文本：

```text
如果系统里还有状态停留在“处理中”超过三十分钟的单据，就提醒我联系李雷一起排查；
明天下午两点给王总发一封邮件，主题写“...”，正文说明“...”。
```

这些场景需要语义理解能力，包括：

- 条件触发识别。
- 指代消解。
- 邮件主题与正文边界。
- 多动作拆分。
- 执行策略识别。
- 动态目标推导。

如果继续用正则补洞，代码会不断膨胀，且每新增一个样例都可能破坏旧样例。

### 调整方案

将 fallback 降级为“简单句保底”：

```text
LLM 可用
  -> LLM structured output
  -> Pydantic schema 校验
  -> validator/postprocess 通用校验
  -> 输出任务

LLM 不可用 + 简单句
  -> fallback 规则解析
  -> validator/postprocess 通用校验
  -> 输出任务并标注 fallback

LLM 不可用 + 复杂长文本
  -> 不再强行拆任务
  -> 返回 warnings
  -> 页面提示修复 LLM 配置后重试
```

复杂文本判断依据包括：

- 文本长度较长。
- 包含多个分隔符或连接词。
- 包含多个动作线索。
- 包含“如果/收到/当/一旦...就/后/再”等条件或事件触发结构。

### 当前改动

- `parser.parse_text(..., allow_complex=False)`：在主接口 fallback 场景下启用复杂文本保护。
- `is_complex_for_fallback`：判断是否应该拒绝规则 fallback 硬拆。
- 前端展示 `warnings`：不再把警告只藏在原始 JSON 中。
- 保留简单句 fallback：用于基础演示和 LLM 临时不可用时的低风险场景。

### 设计原则

- 不再用正则模拟完整语义理解。
- 正则只用于低风险边界：简单切句、邮箱、手机号、显式时间等。
- 对象、条件、指代、邮件正文边界优先交给 LLM structured output。
- 后处理只做通用校验，不继续补大量具体坏例。

### 后续建议

- 优先修复 OpenRouter `401`，确保复杂文本走 LLM 主路径。
- 引入更严格的 schema，例如 `trigger` 独立于 `schedule`，表达 `time/event/condition/manual`。
- 为 LLM 输出增加 `source_span`，让每个字段能追溯到原文片段。
- fallback 保持小而稳，只覆盖短句演示，不追求复杂长文本能力。

## 3. OpenRouter Key 未正确配置会导致复杂文本无法解析

### 问题现象

页面提示：

```text
LLM HTTP 调用失败: 401
文本包含多任务、条件或长上下文；规则 fallback 为避免误拆，未生成任务。
```

此时右侧不会生成任务卡片，只显示解析警告。

### 原因分析

`401` 表示 OpenRouter 鉴权失败。常见原因：

- `.env` 仍然使用示例占位值 `replace-with-your-new-openrouter-key`。
- API Key 已删除、过期或复制不完整。
- 服务启动后才修改 `.env`，但 Python 进程没有重启，仍在使用旧配置。
- Key 曾经暴露在聊天或截图里，被重新生成/废弃后本地没有同步。

当前 demo 已将占位 key 视为“未配置”，避免继续向 OpenRouter 发送无效请求。

### 处理方式

1. 在 OpenRouter 后台重新生成新的 API Key。
2. 修改：

```text
D:\workspace\demo\python-service\.env
```

配置：

```text
OPENROUTER_API_KEY=新的 OpenRouter Key
OPENROUTER_BASE_URL=https://openrouter.ai/api/v1
OPENROUTER_MODEL=deepseek/deepseek-v4-flash:free
```

3. 重启 Python 服务。
4. 再重新点击“生成任务预览”。

### 注意事项

- 不要把真实 API Key 写入代码或提交到仓库。
- 如果 Key 出现在聊天、截图或日志里，应立即在 OpenRouter 后台删除并重新生成。
- 复杂长文本必须走 LLM 主路径；Key 未配置时 fallback 只处理简单句，不再强行拆复杂任务。
