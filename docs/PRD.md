## 2. 产品定位

本 demo 用于验证“AI 秘书把中文自然语言拆成可确认、可调度、可执行任务计划”的核心闭环。

用户输入一段中文自然语言后，系统应识别其中的任务意图，拆分成结构化任务动作，展示给用户确认；用户确认后，系统根据任务是否带有时间计划，立即模拟执行或进入调度队列。

本 demo 重点证明以下能力：

- 多任务自然语言拆解：一段话中包含多个动作时，拆成多个 `TaskAction`。
- 语义字段边界：区分动作类型、目标对象、正文内容、时间计划、置信度和确认要求。
- 人工确认闭环：发送邮件、发消息、回复消息等高风险动作必须先进入预览确认。
- Java + Python 服务边界：Python 负责语义解析，Java 负责业务入口、持久化、确认、编排、调度和模拟执行。
- 可演进架构：后续可替换真实 LLM、真实执行器、正式调度和正式数据库。

## 3. 背景与问题

真实办公场景中，用户经常用口语描述多个待办事项，例如：

> 明天下午三点提醒我给王总发邮件，内容是项目方案已经更新；另外每周五上午十点通知团队群提交周报。收到李雷的消息后回复他：我下午会确认合同。

传统系统要求用户逐项填写收件人、内容、时间、动作类型，成本较高。AI 秘书 demo 希望用自然语言作为入口，让系统先生成可审阅的任务计划，再由用户确认执行。

## 4. 用户与场景

### 4.1 目标用户

- 办公用户：希望用一句话创建提醒、待办、邮件、消息、回复等任务。
- 产品/演示人员：需要展示 AI 秘书从自然语言到可执行计划的链路。
- 研发人员：需要验证 Java 主控服务和 Python 语义能力服务的职责边界。

### 4.2 核心场景

- 用户输入单个任务：如“明天上午十点提醒我整理项目进度”。
- 用户输入多个任务：如“通知团队提交周报，并给王总发邮件同步项目方案”。
- 用户输入带时间任务：一次性时间进入一次性调度，周期性时间生成 cron。
- 用户输入高风险动作：邮件、消息、回复类任务先预览，确认后才模拟执行。
- Python LLM 不可用：回退到本地规则解析器，并在 `warnings` 或 `analysisNote` 中提示。

## 5. 产品目标

### 5.1 当前版本目标

- 复杂条件触发任务的真实外部数据查询。
- 多租户、生产级审计、失败重试、幂等和告警。
- 移动端或完整 PC 客户端体验。

## 6. 功能需求

### 6.1 自然语言任务预览

用户在页面输入一段中文自然语言，点击“生成任务预览”后，Java 服务调用 Python 语义服务，返回任务计划。

输入字段：

- `text`：用户自然语言，必填。
- `timezone`：时区，默认 `Asia/Shanghai`。
- `userId`：用户标识，demo 默认 `demo-user`。

输出字段：

- `planId`：任务计划 ID，由 Java 生成。
- `traceId`：链路追踪 ID。
- `sourceText`：原始输入文本。
- `status`：计划状态，预览后为 `WAITING_CONFIRM`。
- `tasks`：结构化动作列表。
- `warnings`：解析警告。
- `createdAt` / `updatedAt`：创建和更新时间。

验收标准：

- 输入有效中文文本后，页面展示至少一个任务卡片或解析警告。
- 返回结果包含 `planId`、`traceId`、`tasks`。
- 每个任务至少包含动作类型、标题、内容、目标、时间计划、状态和置信度。

### 6.2 多任务拆解

系统应把一段话中的多个动作拆成多个任务。

支持拆分线索：

- 中文句号、分号、换行。
- “另外”“然后”“并且”“同时”等连接词。
- LLM 结构化输出中的多任务数组。

验收标准：

- 输入包含多个动作时，返回多个 `TaskAction`。
- 每个 `TaskAction.sourceSentence` 能对应原始任务片段。
- 拆解后不同任务的目标、内容、时间不应互相串扰。

### 6.3 动作类型识别

当前支持以下 `actionType`：

| 动作类型 | 含义 | 示例 |
| --- | --- | --- |
| `send_email` | 发送或草拟邮件 | “把总结邮件发给王总” |
| `send_message` | 发送消息或通知 | “通知研发群提交周报” |
| `reply_message` | 回复收到的消息 | “收到李雷消息后回复他...” |
| `reminder` | 创建提醒 | “明天三点提醒我确认合同” |
| `create_todo` | 创建待办 | “帮我创建一个跟进合同的待办” |
| `schedule_task` | 定期或计划任务 | “每周五检查周报提交情况” |

验收标准：

- 任务动作类型必须落在上述枚举内。
- 高风险发送/回复类动作 `requiresConfirmation=true`。
- 无法确定动作时，不生成伪任务，并通过警告说明。

### 6.4 目标对象与内容边界

系统必须区分接收对象和任务内容。

目标对象规则：

- 人名、群名、邮箱、动态条件集合可作为目标对象。
- 动词和任务正文不能进入 `target.name`。
- 没有明确外部接收对象时，使用 `unknown` 或 `self`。
- 条件产生的人群，如“未提交周报的人”，使用 `dynamic`。

示例：

| 用户输入 | 目标对象 | 内容 |
| --- | --- | --- |
| 提醒我给李雷确认合同盖章 | 李雷 | 确认合同盖章 |
| 提醒我整理本月项目进度 | unknown/self | 整理本月项目进度 |
| 通知研发群提交周报 | 研发群 | 提交周报 |
| 如果有人下午三点还没提交周报，再提醒他一次 | 未提交周报的人 | 如果有人下午三点还没提交周报，再单独提醒一次 |

验收标准：

- `target.name` 不包含“确认、整理、提交、同步、回复、处理、盖章”等动作词组成的正文。
- 收件人缺失时不强行猜测，返回 `unknown` 并提示人工确认。
- 代词“他/她/他们/对方”不能单独作为目标名称。

### 6.5 时间识别与调度

时间计划结构由 Python 输出，Java 根据计划决定执行方式。

时间类型：

- `none`：无明确时间，确认后立即模拟执行。
- `once`：一次性时间，输出 ISO-8601 `runAt`。
- `recurring`：周期时间，输出 Quartz cron。

验收标准：

- 无时间任务确认后状态变为 `EXECUTED`。
- 一次性时间任务确认后状态变为 `SCHEDULED`，到点后由 Java 定时扫描变为 `EXECUTED`。
- 周期任务确认后状态变为 `SCHEDULED`。
- 时间格式错误时任务状态变为 `FAILED`，并写入执行说明。

### 6.6 用户确认与执行编排

预览后的计划状态为 `WAITING_CONFIRM`。用户点击确认后，Java 服务逐个处理任务：

- 无计划时间：调用模拟执行器，状态变为 `EXECUTED`。
- 一次性或周期计划：进入调度队列，状态变为 `SCHEDULED`。
- 异常任务：状态变为 `FAILED`。

验收标准：

- 未确认的计划不会执行。
- 重复确认非 `WAITING_CONFIRM` 计划时，不重复执行，只返回当前状态摘要。
- 确认接口返回执行摘要：已执行数、已调度数、失败数。

### 6.7 查询能力

系统提供计划查询能力：

- 根据 `planId` 查询单个任务计划。
- 查询全部任务计划列表。

验收标准：

- 查询详情返回完整 `TaskPlan`。
- 查询列表返回当前持久化仓库中的所有计划。
- 不存在的 `planId` 返回错误响应。

### 6.8 前端页面

页面用于 demo 演示，不承载完整产品交互。

页面能力：

- 输入自然语言。
- 生成任务预览。
- 确认并编排。
- 展示任务卡片。
- 展示原始 JSON。

验收标准：

- 页面可从 Java 服务 `http://127.0.0.1:10000` 访问。
- 预览后确认按钮可用。
- 任务卡片展示标题、动作类型、目标、目标类型、计划、置信度、状态、内容、解析备注和执行说明。

## 7. 服务边界

### 7.1 Python 语义服务

职责：

- 接收自然语言文本。
- 调用 LLM 进行结构化解析。
- LLM 不可用时使用本地规则 fallback。
- 使用 Pydantic 校验输出结构。
- 执行后处理，修正目标对象与内容边界。
- 返回固定 JSON Schema。

接口：

- `GET /api/v1/health`
- `POST /api/v1/semantic/parse`

### 7.2 Java 主控服务

职责：

- 提供对外业务接口和静态 demo 页面。
- 生成 `traceId` 和 `planId`。
- 调用 Python 语义服务。
- 保存任务计划和任务动作。
- 提供预览、确认、查询接口。
- 执行立即任务的模拟动作。
- 定时扫描一次性到期任务。

接口：

- `POST /api/ai-task/preview`
- `POST /api/ai-task/{planId}/confirm`
- `GET /api/ai-task/{planId}`
- `GET /api/ai-task`

## 8. 状态流转

```text
WAITING_CONFIRM
  -> CONFIRMED
  -> EXECUTED
  -> SCHEDULED
  -> FAILED
  -> CANCELLED
```

状态说明：

| 状态 | 含义 |
| --- | --- |
| `WAITING_CONFIRM` | 计划已生成，等待用户确认 |
| `CONFIRMED` | 用户已确认计划 |
| `EXECUTED` | 任务已模拟执行 |
| `SCHEDULED` | 任务已进入调度 |
| `FAILED` | 执行或调度失败 |
| `CANCELLED` | 任务取消，当前 demo 预留 |

## 10. 演示用例

### 10.1 多任务混合输入

输入：

```text
明天下午三点提醒我给王总发邮件，内容是项目方案已经更新；另外每周五上午十点通知团队群提交周报。收到李雷的消息后回复他：我下午会确认合同。
```

期望：

- 至少拆出 3 个任务。
- 邮件任务识别为 `send_email` 或提醒发送邮件类任务，目标为王总，内容为项目方案已经更新。
- 周报任务识别为 `send_message` 或 `schedule_task`，目标为团队群，周期为每周五上午十点。
- 回复任务识别为 `reply_message`，目标为李雷，内容为“我下午会确认合同”。

### 10.2 个人提醒

输入：

```text
明天上午九点提醒我整理本月项目进度。
```

期望：

- 动作类型为 `reminder`。
- 目标为 `self` 或 `unknown`，不应把“整理本月项目进度”识别为目标。
- 内容为“整理本月项目进度”。
- 时间为一次性任务。

### 10.3 群通知

输入：

```text
每周五上午十点通知研发群提交周报。
```

期望：

- 动作类型为 `send_message` 或 `schedule_task`。
- 目标为 `group`，名称为“研发群”。
- 内容为“提交周报”。
- 时间为周期任务，cron 表达每周五上午十点。

### 10.4 动态目标

输入：

```text
如果有人到下午三点还没提交周报，再单独提醒他一次。
```

期望：

- 目标类型为 `dynamic`。
- 目标名称类似“未提交周报的人”。
- 不把“他”作为具体目标名称。
- 需要用户确认或人工复核。

### 10.5 LLM 不可用 fallback

前置条件：

- 未配置 `OPENROUTER_API_KEY` 或 LLM 调用失败。

期望：

- Python 服务返回规则解析结果。
- `warnings` 或 `analysisNote` 中体现 fallback 原因。
- Java 预览接口仍可返回 `TaskPlan`。

## 11. 非功能需求

- 可用性：Python 未配置 LLM 时，demo 仍可通过规则 fallback 完成基础演示。
- 稳定性：LLM 输出必须经过 Pydantic 校验和后处理，不能直接信任。
- 可追踪性：每次预览生成 `traceId` 和 `planId`。
- 安全性：高风险动作默认需要确认；demo 不真实触达外部渠道。
- 可维护性：保持 Java/Python 输出契约稳定，避免随意变更字段名。
- 本地开发：默认使用 `Asia/Shanghai` 时区。
